### **Run 2: Deep Technical Specification**

### **1. GRAPH-BASED MEMORY + ORCHESTRATION (7/7 Consensus)**
**(a) SPECIFIC Best Implementation:** **LangGraph** for orchestration + **Mem0** (previously DateTime-Memory) for the graph-based memory store. LangGraph provides the stateful, cyclic graph controller. Mem0 provides the persistent, entity-relationship graph store with embedded vector search.
*   **LangGraph Repo:** `https://github.com/langchain-ai/langgraph`
*   **Mem0 Repo:** `https://github.com/mem0ai/mem0`
*   **Rationale:** LangGraph's `StateGraph` is the de facto standard for defining multi-agent workflows with cycles and shared state. Mem0's explicit `nodes` (entities), `edges` (relationships), and `facts` (observations) model a true knowledge graph. Its hybrid search (vector + keyword + graph traversal) is production-ready.

**(b) EXACT Data Structures & Algorithms:**
*   **Data Structures:**
    *   **Memory Graph:** Adjacency list stored in a graph DB (Mem0 uses `NetworkX`-like in-memory by default, persists to `SQLite`/`Postgres`).
        ```python
        # Simplified Mem0 internal representation
        graph = {
            "nodes": {
                "entity_id_1": {"name": "User", "type": "person", "embedding": [0.1, ...]},
                "entity_id_2": {"name": "ProjectAlpha", "type": "project", "embedding": [...]}
            },
            "edges": {
                ("entity_id_1", "entity_id_2"): {"relation": "WORKS_ON", "weight": 0.95, "timestamp": ...}
            },
            "facts": {
                "fact_hash": {"text": "User prefers Python to Java", "node_ids": ["entity_id_1"], "confidence": 0.87}
            }
        }
        ```
    *   **Orchestration State (LangGraph):** A Pydantic model (`TypedDict` or `BaseModel`) that is passed between nodes (agents/tools).
        ```python
        from typing import TypedDict, List, Annotated
        class AgentState(TypedDict):
            messages: Annotated[List[BaseMessage], add_messages]  # LangChain message history
            memory_facts: List[Dict]  # Retrieved from Mem0
            current_plan: str | None
            task_queue: List[str]
        ```
*   **Algorithms:**
    1.  **Graph-Enhanced RAG:** When a query `q` arrives:
        *   Step 1: Generate an embedding `e_q`.
        *   Step 2: **Hybrid Search:** Execute `(a) Vector similarity search on node/edge embeddings` + `(b) Keyword/lexical search on fact text`.
        *   Step 3: **Graph Traversal:** For top-k vector hits, perform 1-2 hop traversals to gather contextually related entities/facts (PageRank-inspired bidirectional expansion).
        *   Step 4: **Rerank:** Use a cross-encoder (e.g., `BAAI/bge-reranker-large`) to rerank `[query, retrieved_fact]` pairs.
    2.  **Stateful Orchestration (LangGraph):** A finite state machine where edges are conditional. Core algorithm: the `CompiledStateGraph` executes nodes in topological order (or cycles if `condition` returns `True`), passing and mutating the shared `AgentState`.

**(c) INTERACTION WITH OTHER FEATURES:**
*   **Feeds into (2) Declarative Pipelines:** The `AgentState` (with `memory_facts`) is the input context for the pipeline's `Predictor`. The pipeline's output (e.g., a new fact) is written back to Mem0 via a `store_fact` tool.
*   **Feeds into (3) Hierarchical Teams:** The graph memory is the **shared context layer** for all team members. A `Manager` agent queries the graph for team member `skills` (stored as `entity -> relation -> skill` edges) and past `performance` facts.
*   **Feeds into (4) Dual-System Memory:** The **Long-Term Graph Store** *is* the Dual-System's "slow" lattice. The "fast" working memory is the `AgentState`'s `messages` list. Paging between them is triggered by `StateGraph` cycles or token limits.
*   **Feeds into (5) RL Memory Construction:** The RL agent (Feature 5) can modify the graph: adding/merging nodes based on state/action rewards, or adjusting edge `weights` based on retrieval success metrics.

**(d) OPTIMAL HARDWARE-SPECIFIC TECH STACK:**
*   **Android/Termux Tablet:** **Impossible to run natively.** Use as a **thin client** via `LangChain`/`Llama.cpp` API client to a server. If local-only: `Termux` + `Llama.cpp` (Q4_K_M GGUF) + a **lightweight graph library** (`networkx` is pure Python, heavy). *Not recommended for core logic.*
*   **NVIDIA Jetson AGX Orin 64GB:** **Primary inference node.**
    *   **Memory:** Mem0 graph store in `SQLite` (for persistence) with `pgvector`-like HNSW index in RAM (64GB is ample).
    *   **Orchestration:** LangGraph (Python, CPU-bound orchestration logic).
    *   **Embedding/LLM:** `TensorRT-LLM` for `nvidia/Llama-3.1-70B-Instruct` (INT8/FP8) and `BAAI/bge-m3` (INT8). `SentenceTransformers` for smaller models.
    *   **Key Libs:** `langgraph`, `mem0`, `transformers`, `torch`, `trt-llm`, `faiss-gpu` (for fast ANN fallback).
*   **Desktop PC with GPU (e.g., 2x RTX 4090):** **Control plane & heavy model node.**
    *   **Memory:** Mem0 with `Postgres` + `pgvector` (HNSW index). Use `Redis` as a hot cache for graph sub-graphs.
    *   **Orchestration:** LangGraph, potentially parallelized across multiple graphs.
    *   **Embedding/LLM/Reranker:** `vLLM` for high-throughput serving of multiple 70B/128B models. `FAISS-GPU` for massive-scale vector search.
    *   **Key Libs:** `langgraph`, `mem0[postgres]`, `vllm`, `transformers`, `redis`, `pgvector`.

### **2. DECLARATIVE SELF-IMPROVING PIPELINES (7/7)**
**(a) SPECIFIC Best Implementation:** **DSPy** with **Signatures** and **Teleprompters**. The "In-Memory Learning" is achieved via DSPy's `BootstrapFewShot` or `MIPRO` optimizers that *compile* a program (pipeline) using a small set of labeled demonstrations, producing optimized prompts and LM calls.
*   **DSPy Repo:** `https://github.com/stanfordnlp/dspy`
*   **Rationale:** It's the only framework that treats LMs as *compilable modules*, not prompt templates. The `@tool` and `@predictor` abstractions are perfect for declaring agent capabilities.

**(b) EXACT Data Structures & Algorithms:**
*   **Data Structures:**
    *   **Program:** A directed acyclic graph (DAG) of `Predictor` (LM) and `Tool` (function) modules, connected by variable bindings.
    *   **Signature:** A typed interface (input/output fields) for a module. Stored as a Pydantic-like schema.
        ```python
        class GenerateAnswer(dspy.Signature):
            context = dspy.InputField(desc="May be relevant information.")
            question = dspy.InputField()
            answer = dspy.OutputField()
        ```
    *   **Teleprompter State:** A dictionary mapping `module_name -> (demo_set, prompt_template, LM_module)`.
*   **Algorithms:**
    1.  **Compilation (BootstrapFewShot):** For each module `M` in the pipeline:
        *   For each demo `(input, gold_output)` in a small labeled set (`demos`):
            *   Execute the **entire pipeline** with `input` to get `predicted_output`.
            *   Execute `M` in isolation with its *true input* from the pipeline run to get `module_prediction`.
            *   Evaluate `module_prediction` vs. `gold_output` using a **metric function** (e.g., exact match, LLM-as-judge).
        *   Select top-k `demos` for `M` that most improved the metric across all demos.
        *   Construct a **few-shot prompt** from these `demos` and the `Signature` instructions.
    2.  **In-Memory Learning (on deployment):** The compiled program (with optimized few-shot prompts) is the "learned" state. To adapt online:
        *   Store recently successful `(input, full_pipeline_output)` tuples.
        *   Periodically (or on failure), run a **micro-bootstrap** on new examples, potentially updating the few-shot sets for relevant modules.

**(c) INTERACTION WITH OTHER FEATURES:**
*   **Consumes from (1) Graph Memory:** `context` field in a `Signature` is populated by a **query** to the `Graph Store`. The query is itself a `Predictor` (`GenerateQuery` Signature) that takes the user question and `AgentState` to produce a graph search query.
*   **Executes within (3) Hierarchical Teams:** A `CrewAI` `Task` can be *implemented* by a DSPy `Program`. The `Process` (sequential/hierarchical) maps to the DSPy DAG execution order.
*   **Produces for (4) Dual-System Memory:** The final `answer` (or intermediate `module_prediction`) that is deemed important (by a saliency `Predictor`) is **written** to the `Graph Store` as a new `fact`.
*   **Guided by (5) RL Memory:** The RL agent's `reward` signal can be used to **re-weight the metric** for the DSPy compiler. If a certain pipeline structure yields high reward, the compiler will preferentially select demos that lead to that structure.

**(d) OPTIMAL HARDWARE-SPECIFIC TECH STACK:**
*   **Jetson/Desktop:** Run DSPy's compiler **offline** to produce the optimized, static few-shot prompts. Deploy the **compiled program** (just a JSON/prompt template + tool calls) to inference hardware. The runtime cost is just the LM calls.
*   **Key Libs:** `dspy-ai` (core), `dspy`-`optimizers` (bootstrap, MIPRO). The compiled program is framework-agnostic and can be executed with any LM backend (`openai`, `vllm`, `ollama`).

**(e) CONNECTION IN DIAGRAM:**
The **"Declarative Pipeline Definition"** block sits *inside* the **"Hierarchical Agent Team"** block for a given task. It is *compiled* using a small dataset (which can be stored in the **Graph Memory** as `demos`). At runtime, its `context` input is fetched via a **Graph Query** module from the **Graph Store**. Its output is sent to the **Dual-System Paging** for potential storage.
