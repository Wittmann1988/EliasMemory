## 1️⃣  COMPLETE IMPLEMENTATION PLAN  
**Goal:** Deploy a unified, self‑improving agent swarm on three target platforms (Termux/Android, Jetson AGX Orin 64 GB, Desktop PC).  
The stack that achieved consensus is:

| Layer | Selected Engine / Library | Why it was chosen (consensus) |
|-------|---------------------------|--------------------------------|
| **Core orchestration** | **LangGraph** + **DSPy** | LangGraph gives a deterministic graph‑based runtime; DSPy provides compile‑time prompt optimisation that works across languages. |
| **Memory & state** | **Mem0** (light‑weight vector store) + **sqlite‑vec** | Mem0 gives multi‑turn short‑term recall; sqlite‑vec provides a persistent, SQL‑friendly vector DB for long‑term facts. |
| **Multi‑agent coordination** | **CrewAI** (wrapper) + **AgentFly RL** (meta‑controller) | CrewAI implements the “team” pattern; AgentFly supplies a reinforcement‑learning policy that dynamically re‑weights agents based on reward signals. |
| **Embedding engine** | **NVIDIA Embeddings (2048‑dim)** (GPU‑accelerated) | High‑dimensional dense vectors from a production‑grade CLIP‑like model; works on CUDA‑enabled devices (Jetson & Desktop) and via ONNX on Termux (via `torch` ARM wheels). |
| **Persistence** | **sqlite‑vec** (extension) + **tarfs** (read‑only rootfs for Termux) | Zero‑config SQLite + vector extension is portable; tarfs allows embedding a pre‑populated DB in the Termux app bundle. |
| **Evaluation & Logging** | **Prometheus + Grafana** (local) + **Phoenix** (open‑source tracing) | Gives observability on all three hardware profiles. |

## 2️⃣  EVENT BUS PROTOCOL  
The **Event Bus** is the contract‑first communication layer that decouples agents, the RL controller, and the persistence layer.  

| **Aspect** | **Specification** |
|------------|-------------------|
| **Message type** | Typed `dataclass` defined in `src/bus/events.py`. All events inherit from `BaseEvent`. |
| **Transport** | `aio-pubsub` (asyncio‑compatible) backed by an **in‑memory channel** for intra‑process and a **Redis pub/sub** for cross‑process (used only on Desktop & Jetson). |
| **Message envelope** | ```python\nclass BaseEvent(NamedTuple):\n    event_id: UUID\ntimestamp: datetime\nsource: str   # e.g. "agent_0", "controller", "memory_store"\npayload: dict\n``` |
| **Supported events** | 1. `AGENT_STARTED` – payload `{agent_id, skill_profile}`<br>2. `AGENT_STOPPED` – payload `{agent_id, termination_reason}`<br>3. `MEMORY_UPDATED` –payload `{key, value, ttl}`<br>4. `REWARD_EVAL` –payload `{score, metric_name}`<br>5. `PIPELINE_RECONFIG` –payload `{graph_id, node_weights}`<br>6. `SHUTDOWN` –payload `{grace_seconds: int}` |
| **QoS** | - **At‑most‑once** for state‑changing events (`MEMORY_UPDATED`, `REWARD_EVAL`).<br>- **At‑least‑once** for control events (`PIPELINE_RECONFIG`, `SHUTDOWN`) – broker acknowledges after processing.<br>- **Message TTL** configurable (default 10 s) to prevent stale events. |
| **Schema registration** | `src/bus/event_registry.py` holds a **registry of JSON‑Schema draft‑07** validators. Each consumer registers its expected schema; the bus validates on receipt. |
| **Serialization** | `orjson` (fast, deterministic) for payload; versioned (`event_version: int = 1`). |
| **Flow example** (reward feedback loop):<br>1️⃣ Agent emits `REWARD_EVAL` with `{score: 0.87, metric: "completion_accuracy"}`<br>2️⃣ RL controller subscribes, aggregates scores over a sliding window (5 s).<br>3️⃣ When median score drops < 0.6, controller publishes `PIPELINE_RECONFIG` → LangGraph re‑weights node priorities.<br>4️⃣ All agents receive `AGENT_RECONFIGURED` event to reload their policy. |
| **Testing** | `tests/test_event_bus.py` spins up a **local Redis** container, publishes events, asserts ordering, TTL drop‑off, and schema validation. |

            self._loop.create_task(self._redis_listener())

    async def publish(self, event: BaseEvent) -> None:
        raw = orjson.dumps(event.__dict__).decode()
        if self._redis_url:
            # use aioredis publish
            ...
        else:
            await self._pubsub.publish(event.event_id, raw)

    async def subscribe(self, event_type: type[BaseEvent], callback):
        async def wrapper(raw):
            data = orjson.loads(raw)
            ev = event_type(**data)
            await callback(ev)
        await self._pubsub.subscribe(event_type.__name__, wrapper)

## 3️⃣  SELF‑IMPROVEMENT LOOP DESIGN  

A **four‑stage loop** that runs *continuously* on every hardware profile. The loop is deliberately *stateless* except for persisted vectors (sqlite‑vec).  

1. **Sense** – Each agent emits `REWARD_EVAL` events whenever it completes a user‑defined micro‑task (e.g., “answer a boolean query”, “schedule a calendar event”). The reward is a *scalar* derived from external feedback (e.g., user rating, API success flag, cross‑validation metric).  

2. **Store** – `MemoryStore` (a thin wrapper around Mem0 + sqlite‑vec) writes the tuple `(reward, context_vector, timestamp)` keyed by a UUID. If the reward is *higher than a personal best*, the entry is flagged as a **high‑water mark**.  

3. **Learn** – The **AgentFly RL controller** subscribes to `REWARD_EVAL` and accumulates a sliding‑window reward histogram. When a configurable “learning trigger” fires (e.g., > 100 new events in the last 30 s *or* reward median crosses a threshold), the controller does one of:  

   - **a)** **Policy Update** – Takes the latest reward‑conditioned state (embedding of recent dialogue + memory vector) → feeds into an **ONNX policy network** (trained offline with PPO). The new policy is exported as a small (`~200KB`) ONNX model that updates the agent’s action‑selection function *in‑place* (no restart).  
   - **b)** **Graph Re‑weight** – Calls `EventBus.publish(PIPELINE_RECONFIG(...))`. This message carries a new adjacency matrix where nodes with higher *improvement scores* get a larger out‑degree weight.  

4. **Deploy** – Agents consume `PIPELINE_RECONFIG` and **hot‑reload** their LangGraph node definitions via `dspy.compile()` (DSPy automatically recompiles the prompt optimizer). The updated graph is then executed **asynchronously**, allowing *zero‑downtime* learning.  

### 3.1  Reward Function (example)

```python
def compute_reward(task_output: str, ground_truth: str) -> float:
    # Simple exact‑match reward; can be replaced with similarity model score
    if task_output.strip() == ground_truth.strip():
        return 1.0
    # Semantic similarity via cosine similarity of embeddings (2048‑dim)
    sim = cosine_similarity(embedding(task_output), embedding(ground_truth))
    return 0.7 * sim + 0.3   # bias toward exact matches
```

### 3.2  Safety guardrails  

| Guardrail | Mechanism |
|-----------|-----------|
| **Reward clipping** | `np.clip(reward, -1.0, 1.0)` before publishing. |
| **Maximum policy updates per hour** | A token bucket; after 5 updates the controller forces a 30‑s back‑off. |
| **Model version lock** | The ONNX policy file is version‑hashed (`policy_v{hash}.onnx`). If a newer version is published, the controller validates OSEMU checksum before hot‑swap. |
| **Memory quota** | sqlite‑vec set to keep only the most recent **500k** vectors (LRU). |

## 4️⃣  PRIORITY MATRIX (Difficulty ↔ Impact)

| # | Feature | Difficulty (1‑5) | Impact (1‑5) | Reasoning |
|---|---------|------------------|--------------|-----------|
| 1 | **Unified orchestration (LangGraph + DSPy)** | 3 | 5 | Core runtime; must be stable across CPUs/GPUs. |
| 2 | **Event bus (aio‑pubsub + schema registry)** | 2 | 5 | Needed for every agent‑to‑agent communication; lightweight. |
| 3 | **Memory subsystem (Mem0 + sqlite‑vec)** | 3 | 4 | Persist long‑term facts while staying portable. |
| 4 | **AgentFly RL controller** | 4 | 4 | RL policy training adds complexity; still essential for self‑improvement. |
| 5 | **NVIDIA embeddings on ARM (via ONNX)** | 2 | 3 | Embedding quality drives reward fidelity; easy to ship. |
| 6 | **Agent warm‑swap / hot‑reload** | 3 | 4 | Prevents downtime; requires careful graph versioning. |
| 7 | **CI/CD pipeline for three platforms** | 3 | 5 | Automation reduces human error; high CI impact. |
| 8 | **Observability stack (Prometheus+Grafana+Phoenix)** | 2 | 3 | Nice‑to‑have, but not MVP critical. |
| 9 | **Docker‑based deployment on Jetson** | 4 | 4 | Jetson native images involve JetPack constraints. |
|10 | **Android Termux APK bundling** | 5 | 3 | Packaging challenges but lower user impact. |

**Interpretation:**  
- Highest *priority* (quick wins) = items **1, 2, 3, 7**.  
- Next batch (MVP‑compatible) = **4, 6, 9**.  
- Nice‑to‑have (post‑MVP) = **5, 8, 10**.
